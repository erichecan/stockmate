# Mobigo 类目导入与批发站方案总结

> 文档时间：2026-03-14  
> 背景：客户网站 [Mobigo.ie](https://www.mobigo.ie/register.asp) 为爱尔兰手机/PC 配件批发站，其导航类目导入当前仓库系统作种子数据；批发站面向零售商，与仓库拣货/备货/出库全流程打通。

---

## 一、第一步：抓取导航数据并导入仓库系统（已完成）

### 1.1 做了什么

- **数据来源**：根据 [Mobigo.ie](https://www.mobigo.ie/) 左侧导航结构整理出的类目树（该站需登录后才能看到完整导航，此处按公开可见的一级/二级结构及您提供的类目列表整理）。
- **种子数据文件**：`backend/prisma/seed-data/mobigo-categories.json`  
  - 一级类目：BIG BRANDS、ATTRACTIVE、CASES AND COVERS、TEMPERED GLASS、ACCESSORIES、PARTS  
  - 二级/三级：如 LCD/Batteries/Parts、Mobile Cases By Brand、Screen Protector、Cable/Chargers 等，与 Mobigo 导航一致。
- **导入脚本**：`backend/prisma/seed-mobigo-categories.ts`  
  - 登录当前仓库系统 API（默认 `test-company` + admin 账号）  
  - 按树形结构调用 `POST /categories` 创建类目（含 `name`、`nameEn`、`code`、`parentId`、`sortOrder`）  
  - 若类目 `code` 已存在则跳过，并拉取已有类目树填充 `parentId`，避免重复创建。

### 1.2 如何执行导入

```bash
cd backend
# 确保后端 API 已启动（如 npm run start:dev）
npx tsx prisma/seed-mobigo-categories.ts
# 或
npm run seed:mobigo
```

环境变量（可选）：

- `SEED_API_BASE`：API 根地址，默认 `http://localhost:3001/api`
- `SEED_EMAIL` / `SEED_PASSWORD` / `SEED_TENANT`：登录账号与租户，默认与现有 seed 一致

### 1.3 与现有数据的关系

- 与现有 `prisma/seed.ts` 中的类目（如手机壳、保护膜、充电器等）**并存**：Mobigo 类目使用 `MOBIGO_*` 前缀的 `code`，不会与现有 code 冲突。
- 后续可将商品（SPU/SKU）挂到这些 Mobigo 类目下，或与现有类目做映射，由业务决定。

## 二、第二步：批发站产品形态建议

### 2.1 登录策略建议：**部分可浏览 + 登录看全量与价格**

更推荐采用「**未登录可浏览部分内容，登录后看全量商品与价格**」而不是「全部必须登录才能看」。

理由简要：

| 方式 | 优点 | 缺点 |
|------|------|------|
| **全部必须登录** | 数据封闭、易做权限控制 | 不利于 SEO、新客转化低、难以收集「匿名浏览/加购意图」 |
| **部分可浏览 + 登录看价格/下单** | 利于 SEO、新零售商可先逛再注册、能收集浏览/加购意图 | 需区分「公开/登录」接口与前端展示 |

建议规则：

- **未登录可访问**：  
  - 首页、类目树（与 Mobigo 一致的导航）、部分「展示用」商品列表（如每个类目下前 N 个或精选商品）。  
  - 商品可显示：名称、主图、规格、简要描述；**不显示**：批发价、库存数量、起订量、真实库存状态。
- **登录后**：  
  - 全量类目与商品、批发价、库存、起订量、库存特价/清仓区、销售排行、下单与订单历史。

这样既能把「看到价格/下单」与「真实身份」绑定，又能在未登录侧收集到**浏览、点击类目、加购意向**等行为，用于采购意图分析。

### 2.2 建议具备的功能模块

在「与仓库系统打通」的前提下，建议批发站至少包含：

1. **类目导航**  
   - 与仓库系统类目一致（含本次导入的 Mobigo 类目），树形展示，支持按类目筛选商品。

2. **商品列表与详情**  
   - 列表：类目筛选、关键词搜索、排序（上新、销量、价格）。  
   - 详情：规格、多图、描述；登录后显示批发价、库存、起订量、加入购物车/下单。

3. **销售排行 / 畅销品**  
   - 按销量（或按订单行数量）从仓库/订单数据统计，在首页或类目页展示「Top Seller」等，引导采购并反哺选品。

4. **库存特价 / 清仓区**  
   - 与仓库的「库存状态/促销价」或「清仓标记」联动，例如：高库存、临期、清仓款单独入口，刺激消化库存。

5. **购物车与下单**  
   - 登录后：加车、改数量、起订量校验、提交订单；订单进入当前仓库系统，驱动拣货、备货、出库（您已描述的全流程）。

6. **注册/登录与基础账户**  
   - 零售商免费注册、登录后即「下游客户」，与仓库侧客户/订单关联，便于做采购意图与复购分析。

### 2.3 技术实现要点（与现有系统对齐）

- **类目**：直接使用当前仓库系统的 Category API（含本次导入的 Mobigo 类目）；前端树形展示可从 `GET /categories?tree=true` 获取。
- **商品与库存**：与现有 Product/SKU 及库存服务对接；价格使用 `wholesale_price`，库存从现有库存接口获取。
- **订单**：批发站下单生成订单写入当前仓库系统，与现有出库流程一致，无需重复造轮子。
- **权限**：  
  - 未登录：仅开放类目 + 部分商品展示（无价格/无真实库存）的只读接口。  
  - 登录：现有 JWT + tenant/user 体系，控制「全量商品、价格、库存、下单」等接口。

## 三、用「采购意图」反哺仓库与进口

### 3.1 可收集的采购意图数据

在「部分可浏览 + 登录看全量」的前提下，可以同时拿到：

- **未登录侧（匿名）**：  
  - 类目点击、商品点击、加购意向（如「想采购」按钮、收藏）、搜索词、停留时长。  
  - 可用于：哪些类目/单品更受关注、哪些关键词/型号被反复搜索。
- **登录后**：  
  - 真实下单、复购频次、客单价、常采类目与品牌、未下单但多次加车/询价等。  
  - 可用于：谁在采什么、哪些是「高意向未转化」、哪些款应优先备货或清仓。

### 3.2 如何反向影响仓库与进口

- **选品与进口量**  
  - 统计：按类目/品牌/单品的「浏览量、加购意向、搜索次数」与「实际销量」做对比。  
  - 高意向 + 高销：加大进口与安全库存；高意向 + 低销：排查价格/起订量/到货周期。  
  - 低关注款：减少进口或列入清仓，避免压库。

- **安全库存与补货**  
  - 用批发站订单与历史销量预测需求，驱动仓库侧安全库存、补货点与采购单（您已有仓库与采购流程时，可在此接入）。

- **清仓与促销**  
  - 将「库存特价/清仓区」与仓库实际库存与成本挂钩，把滞销、高库存款推到批发站前台，既消化库存又验证需求。

- **新品与试销**  
  - 新品在批发站做「预售/试销」或限量可见，用点击与订单量决定是否大规模进口。

整体闭环：**零售商在批发站的行为与订单 → 汇总为采购意图与销量 → 指导仓库补货、进口数量与选品**。
